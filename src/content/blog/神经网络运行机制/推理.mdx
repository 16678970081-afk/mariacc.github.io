## 推理显存占用

+ 推理时主要的显存占用：
    + 模型权重（Weights）：静态固定。一旦加载，大小不再变化。
    + KV Cache（键值缓存）：持续增长。随着对话轮数增加，随着上下文长度增长而线性扩张，显存占用线性上升。
    + 激活值（Activations）：瞬时波动。只在计算当前层时达到峰值，算完即焚。

当对话轮数过长，总 Token 数达到预设的 max_model_len（如 32k）时，推理引擎将无法再为新的 Token 分配 KV Cache 空间。此时，系统通常会根据配置采取以下动作之一：
停止生成：直接截断输出。
滚动窗口/滑窗：丢弃最早的 Token，腾出空间给新 Token。
报错触发：在 API 调用层面返回 length 相关的终止状态。
----确定一下vllm的策略

在 LLM 推理服务器中，每个独立的会话（Session/Request）都有自己独立的上下文。系统会为每个活跃的会话维护一份专属的 KV Cache。总的 KV Cache 占用确实是所有当前正在处理（Active）的会话上下文占用之和。

为什么必须按会话隔离 KV Cache？
本质问题：Transformer 具有“因果性”，每个会话的对话历史（如：用户 A 聊航天，用户 B 聊烹饪）产生的 Key 和 Value 向量是完全不同的。
后果：如果会话之间不进行显存隔离或错误共享，会导致严重的数据串扰（A 看到 B 的答案）或模型逻辑崩溃。

如何处理“满员”情况？
当 A800 的 80GB 被多个会话填满时，有几种处理策略：

策略 1：排队 (Queueing)：
    做法：新会话必须等旧会话结束并释放显存后才能开始。
    分析：最安全，但延迟高。
策略 2：交换 (Swapping/Offloading)：
    做法：将暂时不说话的会话 KV Cache 搬到 CPU 内存（内存大但慢）。
    分析：适合多会话长停顿场景，但搬回显存时会有明显卡顿。
策略 3：重计算 (Re-computation)：
    做法：丢弃旧会话 KV，下次需要时重新跑一遍前向传播。
    分析：极度浪费算力，通常不作为首选。
